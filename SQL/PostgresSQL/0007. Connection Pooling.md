# 🏊 PostgreSQL Connection Pooling

## 🎯 What is Connection Pooling?

A **connection pool** is a collection of already-open PostgreSQL connections that applications can reuse.

Instead of:

```text
Request 1
 ↓
Open Connection
 ↓
Run Query
 ↓
Close Connection

Request 2
 ↓
Open Connection
 ↓
Run Query
 ↓
Close Connection
```

We do:

```text
Request
   ↓
Borrow Connection
   ↓
Run Query
   ↓
Return Connection
```

This is much faster.

---

# 🤔 Why is Pooling Needed?

In PostgreSQL, every connection creates:

```text
🧠 Memory
+
⚙️ OS Process
+
🔐 Authentication
```

Each connection is expensive.

Example:

```text
1000 users
      ↓
1000 DB Connections
      ↓
🚨 Database Overloaded
```

# Interview Answer 🎤

> PostgreSQL connection pooling is a technique where a fixed number of database connections are created and reused by multiple application requests. Instead of opening and closing connections for every request, applications borrow connections from a pool and return them after use. Pooling improves performance, reduces memory consumption, and increases scalability. Common pooling solutions include PgBouncer, Pgpool-II, and application-level pools such as SQLAlchemy and HikariCP. In serverless environments like Cloud Run, connection pooling is critical to prevent PostgreSQL from being overwhelmed by large numbers of concurrent connections.
