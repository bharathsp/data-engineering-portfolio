### üîç What is the **Frame Clause** in SQL Window Functions?

The **`FRAME CLAUSE`** in SQL defines **which rows** are included in the window for a function like `SUM()`, `AVG()`, `FIRST_VALUE()`, `LAST_VALUE()`, `NTH_VALUE()` **relative to the current row**.

---

### ‚úÖ **General Syntax**

```sql
<window_function>() OVER (
  PARTITION BY ...
  ORDER BY ...
  ROWS | RANGE BETWEEN <start> AND <end>
)
```

---

### üìö Frame Types

| Keyword  | Description                            |
| -------- | -------------------------------------- |
| `ROWS`   | Based on **physical row positions**    |
| `RANGE`  | Based on **values in ORDER BY column** |
| `GROUPS` | Based on peer groups (rarely used)     |

---

### ‚è≥ Frame Boundaries

| Boundary              | Meaning                                 |
| --------------------- | --------------------------------------- |
| `UNBOUNDED PRECEDING` | From the **first row** in the partition |
| `CURRENT ROW`         | The **current row**                     |
| `N PRECEDING`         | N rows **before the current row**       |
| `N FOLLOWING`         | N rows **after the current row**        |
| `UNBOUNDED FOLLOWING` | Up to the **last row** in the partition |

---

Absolutely! Let‚Äôs walk through the **`FRAME CLAUSE`** with sample data, applying it to key window functions like `SUM()`, `FIRST_VALUE()`, `LAST_VALUE()`, and `NTH_VALUE()`.

## 1Ô∏è‚É£ `SUM()` with Default Frame

**Query:**

```sql
SELECT 
  Name, Salary,
  SUM(Salary) OVER (
    ORDER BY Salary
  ) AS RunningTotal
FROM Employees;
```

üí° **Default Frame**:
`ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`

üì§ **Output:**

| Name    | Salary | RunningTotal |
| ------- | ------ | ------------ |
| David   | 2000   | 2000         |
| Bob     | 3000   | 5000         |
| Eva     | 3500   | 8500         |
| Alice   | 4000   | 12500        |
| Charlie | 5000   | 17500        |

‚úÖ Explanation: Sums all salaries **up to the current row**.

---

## 2Ô∏è‚É£ `SUM()` with Sliding Frame: Last 2 Rows

**Query:**

```sql
SELECT 
  Name, Salary,
  SUM(Salary) OVER (
    ORDER BY Salary
    ROWS BETWEEN 1 PRECEDING AND CURRENT ROW
  ) AS SlidingTotal
FROM Employees;
```

üì§ **Output:**

| Name    | Salary | SlidingTotal |
| ------- | ------ | ------------ |
| David   | 2000   | 2000         |
| Bob     | 3000   | 5000         |
| Eva     | 3500   | 6500         |
| Alice   | 4000   | 7500         |
| Charlie | 5000   | 9000         |

‚úÖ Explanation: Sum includes **current row + 1 previous row**.

## 3Ô∏è‚É£ `FIRST_VALUE()` with Default Frame

**Query:**

```sql
SELECT 
  Name, Salary,
  FIRST_VALUE(Salary) OVER (
    ORDER BY Salary
  ) AS FirstSalary
FROM Employees;
```

‚ö†Ô∏è **Default Frame**: Same as before
Output:

| Name    | Salary | FirstSalary |
| ------- | ------ | ----------- |
| David   | 2000   | 2000        |
| Bob     | 3000   | 2000        |
| Eva     | 3500   | 2000        |
| Alice   | 4000   | 2000        |
| Charlie | 5000   | 2000        |

‚úÖ Explanation: Works as expected because it uses all rows from start.

---

## 4Ô∏è‚É£ `LAST_VALUE()` with Default Frame

**Query:**

```sql
SELECT 
  Name, Salary,
  LAST_VALUE(Salary) OVER (
    ORDER BY Salary
  ) AS LastDefault
FROM Employees;
```

üì§ **Output:**

| Name    | Salary | LastDefault |
| ------- | ------ | ----------- |
| David   | 2000   | 2000        |
| Bob     | 3000   | 3000        |
| Eva     | 3500   | 3500        |
| Alice   | 4000   | 4000        |
| Charlie | 5000   | 5000        |

‚ö†Ô∏è **Misleading!** You expect it to always be 5000 (last row), but:

> **`LAST_VALUE()` returns the last value in the frame**, and the default frame ends at the **current row**, not the last.

---

## ‚úÖ Fix `LAST_VALUE()` with Full Frame

```sql
SELECT 
  Name, Salary,
  LAST_VALUE(Salary) OVER (
    ORDER BY Salary
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS LastFixed
FROM Employees;
```

üì§ **Output:**

| Name    | Salary | LastFixed |
| ------- | ------ | --------- |
| David   | 2000   | 5000      |
| Bob     | 3000   | 5000      |
| Eva     | 3500   | 5000      |
| Alice   | 4000   | 5000      |
| Charlie | 5000   | 5000      |

‚úÖ Now it always shows the **true last row‚Äôs value** in the ordered partition.

---

## 5Ô∏è‚É£ `NTH_VALUE()` Example: 2nd Lowest Salary

```sql
SELECT 
  Name, Salary,
  NTH_VALUE(Salary, 2) OVER (
    ORDER BY Salary
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS SecondLowest
FROM Employees;
```

üì§ **Output:**

| Name    | Salary | SecondLowest |
| ------- | ------ | ------------ |
| David   | 2000   | 3000         |
| Bob     | 3000   | 3000         |
| Eva     | 3500   | 3000         |
| Alice   | 4000   | 3000         |
| Charlie | 5000   | 3000         |

‚úÖ Explanation: Always returns the **2nd value** from the full ordered partition.

| Function        | Default Frame                           | Needs Frame Fix? |
| --------------- | --------------------------------------- | ---------------- |
| `SUM()`         | OK by default                           | ‚ùå                |
| `FIRST_VALUE()` | OK by default                           | ‚ùå                |
| `LAST_VALUE()`  | Needs `UNBOUNDED FOLLOWING`             | ‚úÖ                |
| `NTH_VALUE()`   | Needs full frame for consistent results | ‚úÖ                |

---

### ‚ö†Ô∏è Why Frame Clause Matters

* Without the correct frame, functions like `LAST_VALUE()` or `NTH_VALUE()` may behave **unexpectedly**.
* Frame clauses control **how much data** a window function can "see" for each row.
